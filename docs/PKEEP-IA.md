# PKEEP - 프로젝트 맥락 관리 시스템

## 개요

**PKEEP**(Project Context Keeper)은 프로젝트 진행 과정에서 발생하는 회의, 결정, 구현 등의 맥락을 시각화하고 관리하는 도구입니다. 흩어진 프로젝트 정보를 연결하여 "왜 이렇게 결정했는지", "누가 담당했는지"를 쉽게 파악할 수 있습니다.

## 핵심 기능 상세

### 1. 결정 (Decisions)

프로젝트의 모든 결정사항과 그 맥락을 시각화합니다.

#### 컨텍스트 흐름 뷰 (Context Flow View)

결정-미팅-구현 간의 연결 관계를 React Flow로 시각화합니다.

| 기능 | 설명 |
|------|------|
| **노드 타입** | 미팅(파랑), 결정(청록), 화면(보라), Github(회색), Slack(주황) |
| **연결 관계** | 생성됨(파랑), 변경됨(주황), 구현됨(초록), 논의됨(보라), 영향(분홍) |
| **상태 표시** | 확정(초록), 검토중(회색), 대체됨(보라), 폐기(빨강), 비활성(슬레이트), 초안(회색) |
| **담당자 정보** | 주 담당자, 참여자, 리뷰어 표시 |
| **태스크 연결** | 결정에 연결된 태스크와 진행률 표시 |
| **태스크 필터** | 특정 태스크 선택 시 관련 노드만 하이라이트 |

#### 상태별 보기 (Decision Status View)

결정을 상태와 위계별로 그룹화하여 관리합니다.

**상태 탭**
| 탭 | 아이콘 | 설명 |
|-----|--------|------|
| 전체 | Layers | 모든 결정 표시 |
| 확정 | CheckCircle | 최종 확정된 결정 |
| 검토중 | AlertTriangle | 리뷰어 승인 대기 중 |
| 초안 | Clock | 아직 확정되지 않은 결정 |
| 대체됨 | History | 새로운 결정으로 대체됨 |
| 폐기 | XCircle | 더 이상 유효하지 않음 |
| 비활성 | Archive | 일시적으로 비활성화 |

**결정 위계 (Hierarchy)**
| 위계 | 설명 |
|------|------|
| **root** | 최상위 결정 (다른 결정에 의존하지 않음) |
| **child** | 하위 결정 (상위 결정에 의존) |
| **revision** | 수정본 (기존 결정의 변경) |

**결정자 역할**
| 역할 | 설명 |
|------|------|
| **Owner** | 결정의 주 담당자, 최종 책임자 |
| **Approver** | 결정 승인 권한자 |
| **Contributor** | 결정에 참여한 기여자 |

### 2. 기각 히스토리 (Rejected Alternatives)

과거에 기각된 대안들을 기록하고, 유사한 제안이 다시 등장할 때 경고합니다.

```
┌─────────────────────────────────────────────────────────────┐
│  기각 히스토리                              [+ 새 기각 안건] │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ 🚫 Redis 캐싱 도입                                    │  │
│  │ ─────────────────────────────────────────────────── │  │
│  │ 기각 사유: 현재 트래픽으로는 과도한 인프라 비용        │  │
│  │ 연결: DEC-001 Next.js 14 사용                        │  │
│  │ 제안자: 박지민 | 기각일: 2024-01-15                   │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ 🚫 GraphQL API 전환                                   │  │
│  │ ─────────────────────────────────────────────────── │  │
│  │ 기각 사유: 팀 러닝커브와 기존 REST API 활용도 고려     │  │
│  │ 연결: MTG-001 프로젝트 킥오프                        │  │
│  │ 제안자: 김철수 | 기각일: 2024-01-10                   │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 충돌 감지 (Conflict Detection)

새로운 결정 생성 시, 기각된 대안과 유사도를 검사합니다.

| 유사도 | 경고 수준 | 동작 |
|--------|----------|------|
| 80% 이상 | 🔴 높음 | 강력한 경고, 기각 사유 표시 |
| 50-79% | 🟡 중간 | 주의 경고, 검토 권장 |
| 50% 미만 | 없음 | 경고 없음 |

**충돌 감지 시 옵션**
- **무시하고 진행**: 경고를 확인하고 결정 생성 계속
- **기각 내역 검토**: 기각 상세 페이지로 이동
- **취소**: 결정 생성 취소

### 3. 보드 (Boards)

다양한 시각화 방식으로 프로젝트 맥락을 탐색합니다.

#### 마인드맵 뷰 (Mindmap View)

D3.js Force-directed 그래프로 전체 노드 관계를 시각화합니다.

```
┌─────────────────────────────────────────────────────────────┐
│  [전체] [프로젝트] [팀] [담당자] [키워드]   [검색...]       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│         ┌──────┐                                            │
│         │ MTG  │ ─────→ ┌──────┐                           │
│         │ -001 │        │ DEC  │ ─────→ ┌──────┐          │
│         └──────┘        │ -001 │        │ PR   │          │
│              │          │  ✓   │        │ #28  │          │
│              │          └──────┘        └──────┘          │
│              ↓               │                             │
│         ┌──────┐        ┌──────┐                          │
│         │ DEC  │        │ SCR  │                          │
│         │ -002 │        │ -001 │                          │
│         │ 대체됨│        └──────┘                          │
│         └──────┘                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**결정 상태별 노드 스타일**
| 상태 | 테두리 색상 | 뱃지 | 투명도 | 효과 |
|------|----------|-----|------|-----|
| 확정 | 에메랄드 | ✓ | 100% | - |
| 검토중 | 황색 | ⏳ | 100% | - |
| 대체됨 | 보라 | → | 50% | - |
| 폐기 | 빨강 | ✕ | 40% | 취소선 |
| 비활성 | 슬레이트 | ⏸ | 50% | 점선 테두리 |
| 초안 | 회색 | 📝 | 70% | 점선 테두리 |

#### 어피니티 뷰 (Affinity View)

포스트잇 스타일 카드로 그룹화하여 정보를 정리합니다.

| 그룹핑 모드 | 설명 |
|------------|------|
| **회의록별** | 각 미팅에서 나온 논의와 결정을 그룹화 |
| **태스크별** | 할 일 / 진행 중 / 완료 상태별 분류 |
| **담당자별** | 담당자가 작업한 항목 그룹화 |
| **주제별** | AI가 UI/UX, 인증, 기술 스택 등으로 자동 분류 |
| **직접 분류** | 사용자가 드래그앤드롭으로 직접 분류 |

#### 칸반 뷰 (Kanban View)

| 컬럼 | 설명 |
|------|------|
| **백로그** | 아직 시작하지 않은 결정/태스크 |
| **검토 중** | 리뷰 대기 중인 항목 |
| **진행 중** | 현재 작업 중인 항목 |
| **완료** | 완료된 항목 |

## 데이터 모델

### Decision (결정)

```typescript
interface Decision {
  id: string
  code: string              // ex: "DEC-001"
  title: string
  content?: string
  date: string

  // 상태 & 위계
  status: DecisionStatus    // draft | pending | confirmed | superseded | deprecated | disabled
  hierarchy: DecisionHierarchy  // root | child | revision

  // 결정자 정보
  owner: DecisionMaker
  approvers?: DecisionMaker[]
  contributors?: DecisionMaker[]

  // 위계 관계
  parentDecisionId?: string     // 상위 결정
  supersededById?: string       // 이 결정을 대체한 새 결정
  supersedes?: string           // 이 결정이 대체한 이전 결정

  // 수정 이력
  revisions?: DecisionRevision[]
  currentVersion: number

  // 메타
  area?: string
  keywords?: string[]
  projectId: string
  meetingId?: string
  affectedScreenIds?: string[]

  createdAt: string
  updatedAt: string
}
```

### DecisionMaker (결정자)

```typescript
interface DecisionMaker {
  id: string
  name: string
  avatar?: string
  role: 'owner' | 'approver' | 'contributor'
  decidedAt?: string
}
```

### RejectedAlternative (기각된 대안)

```typescript
interface RejectedAlternative {
  id: string
  projectId: string
  decisionId?: string        // 어떤 결정에서 기각됐는지
  meetingId?: string         // 어떤 미팅에서 논의됐는지
  title: string              // 기각된 안건 제목
  description?: string       // 상세 내용
  rejectionReason: string    // 기각 사유
  proposedBy?: string        // 누가 제안했는지
  rejectedAt: string         // 기각 날짜
  keywords?: string[]        // 유사도 검색용 키워드
  createdAt: string
}
```

### TimelineItem (타임라인 항목)

```typescript
interface TimelineItem {
  id: string
  type: 'decision' | 'meeting' | 'github' | 'slack' | 'screen'
  category: 'decision' | 'meeting' | 'implementation' | 'screen'
  source: 'manual' | 'figma' | 'github' | 'slack'
  code: string
  title: string
  date: string
  description?: string
  status?: 'confirmed' | 'changed' | 'pending' | 'superseded' | 'deprecated' | 'disabled' | 'draft'

  // 작업자 정보
  owner?: Person
  contributors?: Person[]
  reviewers?: Person[]

  // 태스크
  tasks?: Task[]

  // 연결 관계
  connections: {
    sources: Connection[]   // 출처
    impacts: Connection[]   // 영향
  }
}
```

### 연결 관계 (ConnectionRelation)

| 관계 | 설명 | 색상 |
|------|------|------|
| `created_from` | ~에서 결정됨 | 파랑 |
| `changed_by` | ~로 인해 변경됨 | 주황 |
| `implemented_in` | ~에서 구현됨 | 초록 |
| `discussed_in` | ~에서 논의됨 | 보라 |
| `affects` | ~에 영향 | 분홍 |

## 사용 시나리오

### 시나리오 1: 결정 맥락 파악
> "이 화면이 왜 이렇게 디자인됐지?"

1. **결정** 메뉴 → **컨텍스트 흐름** 탭
2. 해당 화면 노드 클릭
3. 연결된 결정과 미팅 확인
4. 결정의 sources를 따라가며 원래 논의 내용 파악

### 시나리오 2: 결정 상태 관리
> "대체된 결정들 정리해야 하는데..."

1. **결정** 메뉴 → **상태별 보기** 탭
2. **대체됨** 또는 **폐기** 탭 선택
3. 비활성 결정들 목록 확인
4. 필요시 결정 상태 변경

### 시나리오 3: 과거 기각 안건 확인
> "이거 예전에 논의했던 것 같은데..."

1. **기각 히스토리** 메뉴
2. 유사 키워드로 검색
3. 기각 사유 확인
4. 연결된 결정/미팅으로 이동하여 맥락 파악

### 시나리오 4: 새 결정 시 충돌 확인
> "새로운 캐싱 전략 결정하려고 하는데..."

1. **결정** 메뉴 → **새 결정** 버튼
2. 결정 내용 입력
3. 시스템이 자동으로 기각 히스토리와 유사도 검사
4. 충돌 경고 시 기각 사유 검토 후 진행 여부 결정

### 시나리오 5: 담당자별 업무 현황
> "각 팀원이 뭘 하고 있지?"

1. **보드** → **마인드맵** → **담당자** 필터
2. 담당자별 클러스터로 업무 분포 확인
3. 또는 **결정** → **상태별 보기** → **담당자별** 그룹
